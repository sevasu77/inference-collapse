Final Architectureこれもお願い。再設計版

コンテスト提出版では、「Gemma 4 の推論結果をリアルタイムにゲームへ反映する」というアイデアを最短で実現することを優先した。

一方で、継続的な開発や機能追加を考えると、AI・状態管理・ゲームロジック・UI が密結合しており、責務の境界が曖昧になり始めていた。

そのため、プロダクトとして運用可能な構造を目指し、アーキテクチャ全体を再設計した。

各レイヤーの責務
LLM Service

Gemma は推論を行う責務だけを持つ。

ここでは証拠から

Report
Confidence
Severity
Contradiction

を生成する。

重要なのは、

Gemma 自身はゲーム世界を書き換えない。

AIは「世界を制御する存在」ではなく、「状況を解釈するコンポーネント」として位置付けている。

State Manager

AIの推論結果を、そのままゲームへ渡さない。

一度 State Manager が受け取り、

WorldState
CognitiveState
SimulationState
TelemetryState

という責務ごとの状態へ分離する。

これにより

デバッグしやすい
ログが追跡しやすい
将来的な永続化が容易

という利点がある。

Threat Engine

このプロジェクトで最も重要なレイヤー。

LLMの出力をゲーム難易度へ直接反映するのではなく、

「認知情報を物理世界へ変換する」

役割を持つ。

具体的には

Confidence
      ↓
Kinetic Amplification
（敵速度）

Severity
      ↓
Spatial Instability
（座標揺らぎ）

Contradiction
      ↓
Perception Distortion
（視界・FOV）

という変換を行う。

ここが

「AIがゲームを動かす」

ではなく

「AIの認知状態が世界の物理法則へ変換される」

という、本システム独自の設計である。

Simulation Engine

Threat Engine が生成した物理パラメータだけを受け取り、

敵AI
視界
衝突判定
時間
エフェクト

を更新する。

Simulation Engine は

Gemma の存在を知らない。

State の構造も知らない。

渡された物理パラメータだけを処理するため、

AIモデルが変わってもゲームロジックは変更不要となる。

UI Layer

UIは現在の世界を描画するだけであり、

AI推論にもゲームロジックにも介入しない。

これにより

表示変更
Canvasの置き換え
フロントエンドの変更

を他レイヤーへ影響させず実施できる。

この再設計で得られるもの

今回の再設計では、「コードを分割すること」が目的ではない。

目的は、

AI・ゲーム・状態管理・UIの責務を明確に分離し、それぞれを独立して進化できる構造へ移行することである。

これにより、

Gemma以外のLLMへの置き換え
バックエンドのFastAPI化
マルチプレイヤー対応
推論キャッシュ
非同期実行
リプレイ機能
推論ログ解析

といった拡張を、既存コードへの影響を最小限に抑えながら実装できる。
