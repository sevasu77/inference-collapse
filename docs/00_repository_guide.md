# Repository Guide

# はじめに

Inference Collapse のリポジトリをご覧いただきありがとうございます。

このプロジェクトは、単にゲームを作ることを目的としたものではありません。

**「AI の認知状態をゲーム世界の物理法則へ変換する」**

というアイデアを、実際に動くシステムとして検証するための実験プロジェクトです。

そのため、このリポジトリにはコードだけでなく、多くの設計資料が含まれています。

最初は

> 「ファイルが多い…どこから読めばいいの？」

と思うかもしれません。

でも安心してください。

このページは、このリポジトリ全体を案内する **「地図」** です。

---

# まず最初に読むもの

すべて読む必要はありません。

まずは次の順番で読むだけで、このプロジェクトの全体像を理解できます。

```text
📖 README
        ↓
🏗️ 01_architecture.md
        ↓
💡 02_design_decisions.md
        ↓
📜 03_history.md
        ↓
📝 04_lessons_learned.md
        ↓
🚀 05_future_improvements.md
```

ここまで読めば、

* 何を作ったのか
* なぜその設計なのか
* どのように進化したのか
* 今後どう発展するのか

を理解できます。

## 🏗️ 01_architecture.md

### 「設計図」

このプロジェクト全体の構造を説明します。

ここを読むと、

* どんな部品で構成されているのか
* データがどう流れるのか
* 各コンポーネントは何を担当するのか

が分かります。

家でいう「設計図」です。

---

## 💡 02_design_decisions.md

### 「なぜそう設計したの？」

設計には必ず理由があります。

例えば、

* なぜ State を分割したのか
* なぜ LLM を直接ゲームへ接続しないのか
* なぜ Decision Layer が必要なのか

など、設計の背景を説明しています。

---

## 📜 03_history.md

### 「進化の記録」

このプロジェクトは最初から現在の形だったわけではありません。

```
動くプロトタイプ
        ↓
問題発見
        ↓
改善
        ↓
現在
```

という流れで少しずつ進化してきました。

その過程を記録しています。

---

## 📝 04_lessons_learned.md

### 「学んだこと」

開発中に得られた教訓をまとめています。

例えば、

* LLM の出力は必ず検証する
* State は肥大化しやすい
* UI とゲームロジックは分離した方がよい

などです。

未来の自分や他の開発者へのメモでもあります。

---

## 🚀 05_future_improvements.md

### 「これから作るもの」

将来的にどのようなシステムへ発展させる予定なのかをまとめています。

* FastAPI
* WebSocket
* Multi-Agent
* Replay
* Telemetry

などの構想です。

# ADR フォルダ

```
docs/

└── adr/
```

ADR は

**Architecture Decision Record**

の略です。

難しく聞こえますが、

簡単に言えば

**「設計者の日記」**

です。

開発中には、

> 「どちらの設計が良いだろう？」

という判断を何度も行います。

ADR は、その時の考え方を記録したものです。

完成した設計だけでなく、

「なぜそう決めたのか」

まで残しています。

# src フォルダ

コードはすべてここにあります。

```
src/

├── ai/
├── decision_layer/
├── game_state/
├── simulation_engine/
├── ui/
└── data_models/
```

各フォルダは **1つの役割だけ** を担当します。

---

## 🤖 ai/

LLM と通信する部分です。

Gemma などの AI を呼び出します。

---

## 🛡️ decision_layer/

AI の出力を安全なデータへ変換します。

ゲームを AI の予測不能な出力から守る

**「防波堤」**

の役割を持っています。

---

## 🌍 game_state/

ゲーム世界の状態を保存します。

ここではデータだけを管理し、

ゲームロジックは持ちません。

---

## ⚙️ simulation_engine/

ゲーム世界を動かす心臓部です。

Threat Engine が生成した物理パラメータだけを使って、

世界を更新します。

---

## 🖥️ ui/

画面の表示と、

プレイヤーからの入力を担当します。

---

# なぜ設計書が必要なの？

ソフトウェア開発は、

家を建てることによく似ています。

家を建てるには、

* 設計図
* 配線図
* 水道の図面
* 工事記録

が必要です。

ソフトウェアも同じです。

コードだけでは、

* なぜこの設計なのか
* どこを変更してよいのか
* どこを変更してはいけないのか

が分かりません。

設計書は、

**「どう作ったか」ではなく、**

**「どこを変更すれば安全か」**

を教えてくれる地図なのです。

# 最後に

このプロジェクトで一番伝えたいことは、

**「動くものを作ること」だけではありません。**

まず動くプロトタイプを作り、

そこから責務を整理し、

設計を抽出し、

少しずつ改善していく。

そのプロセスそのものが、このプロジェクトの価値だと考えています。

コードだけでなく、設計資料も合わせて読んでいただくことで、

Inference Collapse の設計思想や開発の過程を、より深く理解していただければ嬉しく思います。
