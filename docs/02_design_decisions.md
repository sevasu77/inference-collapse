# Design Decisions

## 概要

本ドキュメントでは、Inference Collapse のアーキテクチャにおいて採用した主要な設計判断と、その背景にある考え方を説明します。

本プロジェクトでは、「最初から理想的な設計」を目指したわけではありません。

まず動作するプロトタイプを構築し、その後に実装を分析して責務を整理するという **Post-Hoc Architecture** のアプローチを採用しています。

ここでは、「なぜこの構造になったのか」を設計者の視点から記録します。

---

# Design Philosophy

本プロジェクトは、次の設計思想に基づいています。

## Prototype First

最初から完璧な設計を目指すのではなく、まずはコアとなる体験を実現することを優先しました。

コンテスト期間という制約の中では、

- AI が世界へ影響を与える
- LLM の推論がリアルタイムに反映される
- プレイヤーがその変化を体験できる

という最も重要な価値を短期間で実証する必要がありました。

そのため、初期実装では意図的に密結合な構造を採用しています。

---

## Post-Hoc Architecture

プロトタイプ完成後に、

- どこが責務だったのか
- どこが境界になるべきだったのか
- どこが密結合だったのか

を分析し、実装から設計を抽出しました。

つまり、


Prototype
↓
Analysis
↓
Architecture


という流れで設計を構築しています。

本プロジェクトのアーキテクチャは、実装を基に後から意味を与えたものです。

---

# なぜ LLM を分離したのか

LLM は本質的に、

- 非決定的
- 応答時間が一定ではない
- 出力形式が揺らぐ

という特徴を持っています。

一方、ゲームシミュレーションは、

- 決定的
- 毎フレーム一定速度で更新
- 再現性が重要

という全く異なる性質を持っています。

この二つを直接結合すると、

- フレームレートの低下
- デバッグ困難
- 再現性の欠如
- AIの変更がゲーム全体へ影響する

という問題が発生します。

そのため、本プロジェクトでは LLM を「世界を書き換える存在」ではなく、

**認知状態を生成するコンポーネント**

として独立させました。

LLM は推論のみを担当し、その結果をゲームへ直接適用しません。

---

# なぜ Decision Layer を導入したのか

LLM の出力をそのままゲームへ渡すことは危険です。

例えば、

- 型の違い
- 数値範囲の逸脱
- JSON の欠落
- モデルごとの出力差

が発生する可能性があります。

そのため、


LLM
↓
Decision Layer
↓
Game


という構造を採用しました。

Decision Layer は、

- バリデーション
- 型変換
- 正規化
- 値の丸め

を担当する「防波堤」として機能します。

これにより、ゲームロジックは LLM の実装詳細を意識する必要がなくなります。

---

# なぜ State を分離したのか

初期実装では、一つの巨大な State にすべての情報を保存していました。


State
├── World
├── AI Result
├── Physics
├── UI
└── Logs


この構造では、

- God Object 化
- 責務の曖昧化
- 修正の影響範囲の拡大
- テスト困難

といった問題が発生しました。

そこで State を役割ごとに分離しました。


GameState
├── EnvironmentState
├── CognitiveState
├── RuntimeState
└── TelemetryState


それぞれの State はデータのみを保持し、ゲームロジックは持ちません。

これにより、

- 責務が明確になる
- 状態追跡が容易になる
- テストしやすくなる
- 永続化しやすくなる

というメリットがあります。

---

# なぜ Threat Engine を独立させたのか

本プロジェクトの最大の特徴は、

**AI の認知状態を物理法則へ変換すること**

です。

Threat Engine は、


AI State
↓
Physics Parameters


という変換のみを担当します。

例えば、

| Cognitive State | Physics |
|-----------------|----------|
| Confidence | Enemy Speed |
| Severity | Spatial Distortion |
| Contradiction | Visual Distortion |

というように、認知情報を物理パラメータへ翻訳します。

ゲーム難易度を直接変更するのではなく、世界の物理法則を変化させることで、AI とゲームロジックの疎結合を維持しています。

---

# なぜ Simulation Engine は AI を知らないのか

Simulation Engine の責務は、

**物理パラメータに従って世界を更新すること**

だけです。

そのため、

- LLM
- Prompt
- API
- JSON

といった AI 固有の情報を一切扱いません。

Simulation Engine が受け取るのは、


Physics Parameters


のみです。

この設計により、

- LLM を Gemma から別モデルへ変更する
- 推論方式を変更する
- AI を複数導入する

といった変更を行っても、ゲームロジックを修正する必要がありません。

---

# Inference-to-Physics Mapping

本プロジェクトでは、


LLM
↓
Text
↓
Game


という一般的な構造を採用していません。

代わりに、


LLM
↓
Cognitive State
↓
Threat Engine
↓
Physics Parameters
↓
Simulation
↓
Player Experience


という一方向のデータフローを採用しています。

AI はゲームを直接制御する存在ではなく、

**世界をどのように認識しているか**

を表現する存在です。

その認知状態を物理法則へ変換することで、ゲーム世界へ間接的な影響を与えます。

---

# 設計原則

本プロジェクトでは、以下の設計原則を重視しています。

- **Prototype First**  
  まず動作するプロトタイプを構築し、アイデアの実現可能性を検証する。

- **Post-Hoc Architecture**  
  動作する実装から責務を抽出し、設計へ昇華する。

- **Separation of Responsibilities**  
  AI・状態管理・シミュレーション・UI を独立したコンポーネントとして設計する。

- **Loose Coupling**  
  各レイヤーの依存関係を一方向に限定し、保守性と拡張性を高める。

- **Inference-to-Physics Mapping**  
  AI の推論結果を直接ゲームへ反映するのではなく、物理法則へ変換する。

---

# まとめ

Inference Collapse の設計は、最初から完成されたものではありません。

制約のあるプロトタイプ開発から得られた知見を分析し、

- AI
- State
- Physics
- Simulation
- UI

それぞれの責務を明確に分離することで、現在のアーキテクチャへ発展しました。

本プロジェクトの価値は、「動くプロトタイプ」を作ったことだけではなく、

**その実装から設計を抽出し、より保守性・拡張性の高いアーキテクチャへ進化させたプロセスそのもの**

にあります。
