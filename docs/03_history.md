# 03. History — プロトタイプからアーキテクチャへの進化

## Overview

このドキュメントでは、本プロジェクトのアーキテクチャがどのような設計判断を経て現在の形へ進化したかを記録します。

本プロジェクトは、最初から完成された設計を目指したものではありません。

まず **動作するプロトタイプ** を構築し、その後に構造を分析・抽出し、責務を整理しながらアーキテクチャへ発展させるという、段階的なアプローチを採用しています。

---

# 出発点：制約付きプロトタイピング

開発当初には、コンテストという明確な時間的制約がありました。

そのため最も重要だったのは、

> **「AI が世界を書き換える体験」を成立させること**

であり、設計の美しさではありませんでした。

そこで、以下の方針を意図的に採用しました。

- LLM の出力をゲームへ直接反映する
- UI・AI・ゲームロジックを分離しない
- 最短距離で動作するデモを完成させる

つまり、

> **Prototype First**

という思想を優先しました。

---

# 初期アーキテクチャ

初期構造は、実装速度を最優先したシンプルな構成でした。

```text
LLM
   │
Python Backend
   │
Game Logic
   │
UI
```

この構造により、

- AI が世界を変える体験
- リアルタイム推論
- ゲームプレイ

を短期間で実現できました。

一方で、この構造は長期的には大きな課題を抱えていました。

---

# 顕在化した問題

## 1. LLM とゲームロジックの密結合

LLM の出力がそのままゲーム世界へ反映されるため、

- 再現性が低い
- デバッグが難しい
- ノイズとロジックの境界が曖昧

という問題が発生しました。

---

## 2. State の肥大化

すべての情報が一つの State に集約されていました。

```text
State
├── World
├── AI Result
├── Threat
├── Engine Data
├── UI State
└── Logs
```

その結果、

- 責務が不明確
- 修正の影響範囲が広い
- スケールしない

という典型的な **God Object（神オブジェクト）** の状態になっていました。

---

## 3. Threat System の責務が曖昧

Threat System は動作していましたが、

- 難易度制御
- 演出
- 物理法則

という複数の役割を同時に担っていました。

つまり、

> 「何を担当する層なのか」

を明確に説明できない状態でした。

---

## 4. レイヤー分離の欠如

初期構造では、Python Backend がすべてを制御していました。

```text
UI
   │
Python Backend
   │
LLM
Game
State
```

この構造では、

- 変更の影響範囲が大きい
- モジュール単位でテストしにくい
- 将来的な FastAPI 化が難しい

という問題がありました。

---

# ボトルネック分析

分析の結果、特に大きな課題は次の二つでした。

## State が境界になっていない

本来 State は「世界の状態」を保持する役割です。

しかし実際には、

- AI
- Game Engine
- UI

すべてが直接 State を変更していました。

つまり、State が共有メモリとなり、

設計上の境界として機能していませんでした。

---

## LLM が直接世界を書き換えていた

本来 LLM は、

> **世界を解釈する存在**

であるべきです。

しかし初期設計では、

> **LLM がゲーム世界を直接変更していました。**

その結果、

AI の認知とゲーム世界が強く結び付いてしまっていました。

---

# 採用した解決策

プロトタイプを維持しながら、責務を段階的に分離しました。

```text
LLM
   │
Decision Layer
   │
GameState
   │
Threat Engine
   │
Simulation Engine
   │
UI
```

この構造では、

- Decision Layer が LLM 出力を正規化する
- GameState が状態だけを保持する
- Threat Engine が認知状態を物理法則へ変換する
- Simulation Engine が世界を更新する

という役割に整理されました。

---

# State の再構造化

State は責務ごとに分割しました。

```text
GameState
├── Environment
├── Cognition
├── Runtime
└── Telemetry
```

これにより、

- データの意味が明確になる
- 責務が整理される
- 依存関係を最小化できる

ようになりました。

---

# Threat Engine の再定義

Threat Engine は難易度調整機能ではありません。

その役割は、

> **AI の認知状態をゲーム世界の物理法則へ変換すること**

です。

| Cognitive State | Physical Effect |
|-----------------|-----------------|
| Confidence | Enemy Speed |
| Severity | Spatial Noise |
| Contradiction | Visual Distortion |
| Hallucination | Field of View |

Threat Engine は、

**認知と物理世界を接続する変換レイヤー**

として位置付けられています。

---

# 検討した代替案

設計検討の過程では、以下の案も検討しました。

| 代替案 | 採用しなかった理由 |
|---------|-------------------|
| LLM → Game Engine を直接接続 | AI とゲームロジックが密結合になる |
| 巨大な State を維持する | God Object 化し、責務が曖昧になる |
| Engine 内で Threat を計算する | 認知と物理法則の境界が失われる |

---

# 設計思想の変化

最も大きな変化は、

> **「AI がゲームを動かす」**

という考え方から、

> **「AI の認知状態を物理モデルへ変換する」**

という考え方へ移行したことです。

この発想の転換により、

AI・状態管理・シミュレーション・UI の責務を明確に分離できるようになりました。

---

# Conclusion

本プロジェクトは、最初から完成されたアーキテクチャを設計したものではありません。

制約の中で動作するプロトタイプを構築し、

その後、

- 問題を分析し
- 責務を整理し
- 意味を与え
- 段階的にアーキテクチャへ昇華してきました。

本プロジェクトの価値は、

**「動くものを作ること」だけではありません。**

動作するシステムから設計を抽出し、改善し、再構築していくというプロセスそのものも、本プロジェクトの重要な成果です。
