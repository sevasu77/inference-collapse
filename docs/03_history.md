# 03. History

## プロトタイプからアーキテクチャへの進化

このドキュメントでは、本プロジェクトがどのような設計判断を経て現在のアーキテクチャへ発展したのかを記録します。

Inference Collapse は、最初から完成された設計を目指したプロジェクトではありません。

まず「AI の認知状態がゲーム世界へ影響を与える」というコアコンセプトを最短で実現し、その後に構造を分析・整理するという **Prototype First** の考え方を採用しました。

---

# 出発点：コンテストという制約

開発当初はコンテスト期間という明確な時間的制約がありました。

この段階で最も重要だったのは、

> **「AI が世界を書き換える体験」を実際に動かすこと**

であり、設計の美しさではありませんでした。

そのため、

- LLM の出力を直接ゲームへ反映する
- UI・AI・ゲームロジックを分離しない
- 最短距離で動くプロトタイプを完成させる

という判断を意図的に選択しました。

この判断によって、短期間でコアコンセプトを実証することができました。

---

# 初期構成

初期の実装では、多くの責務が一つのアプリケーションへ集約されていました。

```
app.py

├── Streamlit UI
├── Gemma API
├── Runtime State
├── Game Logic
├── Physics
├── Rendering
└── Logging
```

構造としては理想的ではありませんでしたが、

- AI 推論
- リアルタイムゲーム
- Canvas 描画

を短期間で統合するには最も効率的な方法でした。

---

# 顕在化した課題

## 1. 密結合

実装が進むにつれて、各コンポーネントがお互いを直接呼び出す構造になっていきました。

例えば、

- UI がゲーム状態を書き換える
- LLM がゲームロジックへ直接影響する
- ゲームロジックが描画状態まで管理する

といった依存関係が生まれました。

その結果、

- 修正の影響範囲が広い
- モジュール単位でのテストが難しい
- 将来的な FastAPI 化が困難

という問題が明確になりました。

---

## 2. God Object 問題

もう一つ大きな課題だったのが、Runtime State の肥大化です。

当初はすべての情報を一つの State へ保存していました。

```
Runtime State

├── World
├── AI Result
├── Physics
├── UI State
├── Timer
├── Logs
└── Debug Data
```

この構造では、

- 状態の責務が曖昧
- 修正時の影響範囲が予測しづらい
- デバッグが難しい
- スケールしない

という、典型的な **God Object** の問題が発生しました。

---

## 3. LLM とゲームロジックの密結合

当初は LLM の出力をそのままゲームへ反映していました。

```
LLM
    ↓
Game Logic
```

この方法は試作としては成功しましたが、

- AI の出力が毎回変わる
- 再現性が低い
- デバッグが困難
- AI モデル変更時の影響が大きい

という課題を抱えていました。

---

# ボトルネックの分析

プロトタイプを分析した結果、根本的な問題は二つあることが分かりました。

## State が境界として機能していない

本来 State は世界の状態を保持するためのものですが、

実際には AI・UI・ゲームロジックがすべて直接 State を操作していました。

State は「記録」ではなく「共有メモリ」と化していたのです。

---

## AI が世界を直接変更していた

本来 LLM は、

**世界を操作する存在ではなく、世界を認知する存在**

として設計したいと考えていました。

しかし試作段階では、

```
LLM
    ↓
Game World
```

となっており、

認知とゲーム世界が強く結合していました。

---

# 設計の転換

ここで設計思想を見直しました。

AI がゲームを直接制御するのではなく、

**AI の認知状態を一度物理法則へ変換する**

という考え方へ移行しました。

```
LLM
      ↓
Cognitive State
      ↓
Threat Engine
      ↓
Physics
      ↓
Simulation
      ↓
Rendering
```

この構造により、

- AI は「世界を認知する」
- Threat Engine は「認知を物理へ翻訳する」
- Simulation Engine は「物理法則だけを見る」

という責務分離が可能になりました。

---

# Prototype First から Post-Hoc Architecture へ

Inference Collapse の設計は、

最初から完成されたアーキテクチャではありません。

まず動くプロトタイプを構築し、

その後に構造を分析し、

責務を抽出し、

アーキテクチャへ昇華させています。

このアプローチを、本プロジェクトでは

**Post-Hoc Architecture**

と呼んでいます。

---

# このプロジェクトで得られた学び

プロトタイプを通じて得られた最も大きな知見は、

**「設計は最初に完成させるものではなく、動くシステムから抽出できる」**

ということでした。

Prototype First によってコアコンセプトを検証し、

その経験をもとに責務を整理することで、

現在のアーキテクチャへ発展しました。

そのため、本プロジェクトの価値は完成したソフトウェアだけではありません。

**動作するプロトタイプから設計を抽出し、より保守性・拡張性の高い構造へ進化させていくプロセスそのもの**も、このリポジトリの重要な成果だと考えています。
